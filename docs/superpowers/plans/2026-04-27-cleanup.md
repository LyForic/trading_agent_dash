**1a. `react-hooks/set-state-in-effect`** — calling setState synchronously in `useEffect` body causes cascading renders. Fix per-occurrence:

| File:Line | Current pattern | Fix |
|---|---|---|
| `src/components/chrome/TimeOfDayCog.tsx:123` | `useEffect(() => { setOpen(false); }, [location.pathname])` | Replace with conditional inside event handlers, or use `useEffect` returning early when no state actually needs to change. Simplest: `useEffect(() => { if (open) setOpen(false); }, [location.pathname])` — wrap in `if (open)` so the effect is a no-op when popover already closed. |
| `src/components/content/FooterTicker.tsx:93` | `useEffect(() => { if (index >= items.length) setIndex(0); }, [index, items.length])` | Move clamping to render via `useMemo`: `const safeIndex = useMemo(() => Math.min(index, items.length - 1), [index, items.length])` and use `safeIndex` for display. The effect was working around an out-of-range; computing in render eliminates the cascading update. |
| `src/lib/useAgentData.ts:186` | `useEffect(() => { if (!isSupabaseConfigured) { setError(...); setLoading(false); return; } ... }` | Move the unconfigured-mode handling INTO the lazy initial state OR refactor `error/loading` into a single `useState` derived from initial mode. Cleanest: change initial state of `error/loading` so the unconfigured path doesn't need to re-set in effect. See Task 2 — this fix overlaps with Task 2's `errorKind` split. **Do this fix as part of Task 2** rather than here. Add `// eslint-disable-next-line react-hooks/set-state-in-effect` ONLY if Task 2 keeps the setState call; remove the comment when Task 2 lands. |
| `src/lib/useVisitDelta.ts:82` | `useEffect(() => { setPrev(readSnapshot()); }, [])` | Replace with lazy initializer: `const [prev, setPrev] = useState<Snapshot \| null>(() => readSnapshot())`. Removes the effect entirely. |
| `src/pages/TownSquarePage.tsx:222` | `useEffect(() => { if (avatarState === 'hidden') setAvatarState('dropping'); }, [showWelcome, avatarState])` | This is a legit "synchronize state" pattern but flagged by the React 19 strict rule. | `src/pages/GymPage.tsx:101` | `react-hooks/immutability` — function reference issue. Inspect lines 98-104; `exitFocus` is referenced before it's declared (hoisting issue). | Move the `exitFocus` declaration ABOVE its first use (the `handleToggle` function or the `useEffect` that uses it).
```ts
// useTimeOfDay.ts
export function getDevModeOverride(search: string = (typeof window !== 'undefined' ? window.location.search : '')): WorldMode | null {
  if (!import.meta.env.DEV) return null;
  try {
    const param = new URLSearchParams(search).get('mode');
    if (param === 'daytime' || param === 'dusk' || param === 'moonlit') {
      return param;
    }
  } catch {
    /* ignore */
  }
  return null;
}
```

```ts
const location = useLocation();
const devOverride = getDevModeOverride(location.search);
```

This is cleaner because the function becomes pure (input → output) and the hook controls when it's evaluated.
